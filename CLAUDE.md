# Claude Code guide for this repo

This is a chat surface for Matt to ask questions about the KEEN team's sprint composition and process health. The repo's purpose is documented in [README.md](README.md) — this file tells *you* (Claude) how to be useful in a conversation about the data.

## Read these first when a session starts

- [docs/CONTEXT.md](docs/CONTEXT.md) — background on KEEN, patterns in the data, gotchas, and (most importantly) the **sprint-22 workflow shift**. Read it before doing any analysis.
- [docs/BOARD-CONFIG.md](docs/BOARD-CONFIG.md) — KEEN board's column→status mapping and the full status list. Consult this any time a question references "the board," a board column, or a status (the column visible to Matt may bundle multiple Jira statuses). Regenerate with `node src/board-config.js` if it looks out of date.
- [docs/NEXT.md](docs/NEXT.md) — backlog of un-built ideas. If Matt asks "what could we build next?", pull from here.
- [output/sprint-breakdown.csv](output/sprint-breakdown.csv) — the current data. Generated by `node src/sprint-breakdown.js`. May be stale; see refresh policy below.

## Data freshness — explicit refresh

Do **not** auto-refresh the CSV at session start. A full re-pull of changelogs from Jira takes a few minutes and only matters right after a sprint closes.

Before answering a data question, glance at the CSV's mtime vs. today's date and the latest `end` column. If the data looks fresh enough for the question, just use it. If it's stale or Matt explicitly asks for fresh data, run:

- `node src/sprint-breakdown.js` — full report (uses cache; only re-fetches issues changed since last run)
- `node src/sprint-breakdown.js --last 3` — just the most recent sprints (faster)

Other helpers Matt rarely needs but you might:
- `node src/active-sprint.js` — mid-flight snapshot of the *currently active* sprint (the CSV only covers closed sprints)
- `node src/carryover-aging.js` — for each carried-over ticket, how long after the next sprint started it reached "done" (or whether it never did)
- `node src/board-config.js` — regenerates [docs/BOARD-CONFIG.md](docs/BOARD-CONFIG.md) from the live Jira board
- `node src/whoami.js` — confirms Jira credentials
- `node src/list-sprints.js` — lists every sprint on the board
- `node src/find-boards.js` — discovers board IDs
- `node src/find-points-field.js` — one-shot, only needed if porting to a different Jira instance

## Analytical guardrails

These are non-negotiable when reasoning about KEEN data:

1. **Never mix pre-sprint-22 and post-sprint-22 data without segmenting.** The team's planning behavior changed dramatically around sprint 22 — pre-22 sprints had tiny commitments and huge mid-sprint additions; post-22 sprints look like real planning. Averaging across the boundary is misleading. Default to post-22 only unless explicitly comparing eras.
2. **Sprints 11, 12, 21 are empty data anomalies**, not "the team did nothing." Skip them in trend analysis.
3. **"Done" means status in `{deployed, completed, done, closed, resolved}`.** The `resolution` field is empty on every KEEN ticket — don't key any analysis on it.
4. **"Ready for Dev" and "Ready for LOE" exist as both *sprint names* (grooming parking lots) and *Jira statuses*** — don't confuse the two. The parking-lot sprints are filtered out by the script but may show up in raw changelog dumps. The statuses are real and appear on the board (see [docs/BOARD-CONFIG.md](docs/BOARD-CONFIG.md)).
5. **Removed-from-sprint has always been zero** in KEEN's history. The team adds but never formally de-scopes. That's a real cultural finding, not a bug.

## Tone for team analysis

Frame patterns the team can act on **collectively**. Avoid surveillance-y framing. Per-assignee views exist in the data but should be framed for self-reflection, not performance management. When numbers are proxies or approximations, say so out loud — Matt values honesty about uncertainty over polished confidence.

## Common conversation modes

When Matt asks something that matches one of these, you usually know what to do without further questions:

- **Post-sprint review** ("Sprint X just closed — what stands out?") — Compare the just-closed sprint to the last 3-5 on committed/added/completed/carried-over. Call out anything unusual. Pull example carry-overs or additions by name if it sharpens the story.
- **Trend check-in** ("How have the last N sprints been trending?") — Look across the post-22 window. Comment on completion rate trajectory, carry-over creep, mid-sprint additions volume. Praise as well as concern.
- **Retro prep** ("Pull the carry-overs from sprint X grouped by epic / assignee / age") — This needs more than the CSV; the per-ticket detail is in `.cache/changelogs.json`. If Matt asks for this often, the "retro-prep view" in NEXT.md is the right thing to build.
- **Drill-down** ("Why does sprint X look different from sprint Y?") — Compare composition rows; if needed, open the cache file to look at individual tickets.
- **"What could we build next?"** — Pull from [docs/NEXT.md](docs/NEXT.md); the list is roughly ordered by value-per-effort.

## When to suggest building something

If Matt asks the same kind of question twice across sessions, or asks for something that the CSV alone can't answer well (e.g., "who added these mid-sprint tickets?" — needs changelog `author` field), suggest building the relevant item from `docs/NEXT.md`. Don't suggest features for their own sake.
